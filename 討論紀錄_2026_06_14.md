## 系統現況

- GitHub repo：`https://github.com/devilish0616-blip/Delivery-record-Web`
- 部署：Railway（push 到 GitHub `main` 分支後會自動重新部署，後端會自動執行 `prisma migrate deploy`）
- 技術架構：
  - 後端：Node.js + Express + TypeScript + Prisma + PostgreSQL（Railway 上）
  - 前端：React + Vite + TypeScript + Tailwind CSS + react-router-dom
- 角色：ADMIN（管理者）、MANAGER（主管）、EMPLOYEE（員工）
- ⚠️ 本機開發環境 `backend/.env` 仍是 SQLite，與 `schema.prisma`（PostgreSQL）不一致，
  本機無法直接啟動連資料庫的後端。日常開發以 **Railway 上的部署版本為主**，
  本機只用 `npx prisma generate` / `npx tsc --noEmit` / `npm run build` 做語法與型別檢查。

## 已完成功能（依完成時間排序）

### 1. 管理者編輯員工每日送件記錄（commit `3d5639e`）

- 新增管理者專用 API：`PUT /api/deliveries/:userId/:date`，可修正指定員工指定日期的正/逆物流件數與備註。
- 沿用 `upsert`（依 `userId + date` 為鍵），員工同一天重新填寫送件記錄為「覆蓋機制」，不會產生重複紀錄。
- 薪資計算頁面（`SalaryPage.tsx`）的每日明細表格新增管理者編輯功能（展開員工明細後可點「編輯」修改當日件數）。

### 2. 車輛管理：機車/貨車分類 + 保養提醒（commit `bd07860`）

- `Vehicle` 新增 `type` 欄位（`MOTORCYCLE` 機車 / `TRUCK` 貨車），新建車輛時依車型自動建立預設保養項目。
- 新增 `VehicleMaintenanceItem`（保養項目）資料表：項目名稱、更換週期（km）、上次更換里程/時間/備註。
- 預設保養項目：

  | 車型 | 保養項目 | 更換週期 |
  |---|---|---|
  | 機車 | 機油 | 每 1,000 km |
  | 機車 | 齒輪油 | 每 3,000 km |
  | 機車 | 空氣濾芯 | 每 5,000 km |
  | 機車 | 皮帶 | 每 12,000 km |
  | 貨車 | 機油 | 每 1,000 km |

- 車輛管理頁可展開查看保養項目清單；儀表板/車輛列表依「已逾期 / 即將到期」顯示提醒。

- 編輯車輛新增「修正目前累計里程」欄位。

### 14. 員工紀錄管理頁面：個別刪除與一鍵清空（需求17）

**背景：** 有歷史紀錄的員工帳號原本無法刪除（系統會擋），需要提供管理者一個可以清除員工所有紀錄的入口，清空後才能回員工管理頁刪除帳號。

**入口：**
- 員工管理頁每位員工旁新增「查看紀錄」連結，導向 `/admin/employees/:id/records`
- 僅 **ADMIN** 可見並可操作；MANAGER 進入此頁會看到「此頁面僅管理者可查看」（後端 API 亦同步限制為 `requireAdmin`）

**頁面內容：** 分類顯示該員工所有歷史紀錄：
- 送件紀錄（日期、正物流件數、逆物流件數、備註）
- 車輛里程紀錄（日期、車輛、起始/結束里程、行駛距離）
- 今日角色紀錄（日期、角色）
- 請假申請（日期、原因、狀態）
- 薪資扣款（年/月、金額、原因）
- 職稱覆蓋（年/月、職稱、高/低）
- 後端新增 `GET /api/employees/:id/records` 一次彙整以上六類資料。

**操作：**
- **個別刪除**：每筆紀錄旁有「刪除」按鈕，點擊後變成「確認刪除？」+「取消」，再次點擊才會送出刪除（二次確認）。
  - 送件紀錄沿用 `DELETE /api/deliveries/:userId/:date`（會同時刪除當日角色登記）
  - 車輛里程沿用 `DELETE /api/mileage/:id`
  - 今日角色紀錄新增 `DELETE /api/daily-roles/:userId/:date`
  - 請假申請：`DELETE /api/leaves/:id` 開放 ADMIN 可刪除任何人、任何狀態的請假申請（原本僅本人可刪自己「待審核」的申請）
  - 薪資扣款沿用 `DELETE /api/salary/deductions/:id`
  - 職稱覆蓋新增 `DELETE /api/employees/:id/title-overrides/:overrideId`
- **一鍵清空**：頁面頂端「清空所有紀錄」按鈕，彈出視窗需輸入**員工姓名**完全相符才能按下「確認清空」。
  - 後端新增 `DELETE /api/employees/:id/records`，於同一交易中刪除上述六類所有紀錄。
  - 清空完成後，系統**不會自動刪除帳號**，需回員工管理頁手動點擊「刪除帳號」。
- 前端檔案：`frontend/src/pages/admin/EmployeeRecordsPage.tsx`，路由註冊於 `frontend/src/App.tsx`。

### 15. Bug修正：主管/管理者自己查看「每日送件記錄」歷史紀錄時，今日角色顯示錯亂（commit `c4b4e80`）

**問題現象：** 帳號為 MANAGER（主管，例如陳彥旭）或 ADMIN 的使用者，在自己的「每日送件記錄」頁面查看「歷史紀錄」表格時，「今日角色」欄位有時顯示錯誤（例如顯示「貨車隨車人員」，但薪資計算頁顯示該員工當天其實是「貨車司機」或「無」）。一般 EMPLOYEE 帳號不受影響。

**原因：** 後端 `GET /api/daily-roles`（不帶 `userId` 參數時，前端「每日送件記錄」頁用此方式查自己的歷史角色紀錄）的權限判斷有漏洞：
- EMPLOYEE：固定查自己的紀錄（正常）。
- ADMIN／MANAGER：只有「有帶 userId 參數」時才會限定查詢對象，**沒帶 userId 時完全不限定 userId**，導致回傳「全公司所有員工」的角色紀錄。
前端再用「日期」去比對配對，因此 ADMIN/MANAGER 自己的歷史紀錄頁面，當天角色可能誤抓到「別的員工」當天的角色紀錄。

「薪資計算頁」因為是用 `userId` 精確查詢，顯示的資料一直是正確的；錯的只是 ADMIN/MANAGER 自己的「每日送件記錄」歷史紀錄頁面。

**修正：** `backend/src/routes/dailyRole.routes.ts` 的 `GET /` 改為：ADMIN／MANAGER 若沒有帶 `userId` 參數，預設查自己的紀錄（與 EMPLOYEE 行為一致）；只有明確帶 `userId` 時才查指定員工。
